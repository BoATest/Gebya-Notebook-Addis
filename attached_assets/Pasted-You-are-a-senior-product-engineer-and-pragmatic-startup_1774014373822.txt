You are a senior product engineer and pragmatic startup CTO.

I am building a mobile-first sales recording app for Ethiopian shop owners, retailers, wholesalers, and informal sellers. The current product already has manual sale entry screens, but I want to shift the product to a voice-first MVP.

CORE PRODUCT INTENT
- The app should make recording a sale easier than writing in a notebook.
- This is for real sellers in Ethiopia.
- Many users speak Amharic or mixed Amharic + English.
- Product names may be in English, while the rest of the sentence may be in Amharic.
- Sellers may speak casually, incompletely, or in noisy environments.
- The goal is NOT perfect AI.
- The goal is speed, convenience, and collecting real usage data.

STRATEGIC RULES

IMPORTANT MVP DECISION
For Version 1, do NOT try to confidently parse full item breakdowns from voice.
Instead, do this flow:

1. User taps a big microphone button
2. User records a short sale note
3. Audio is sent to speech-to-text
4. App returns:
   - transcript
   - detected_total (best effort, simple logic)
5. App shows:
   - what it heard
   - detected total
   - buttons: Save / Fix / Re-record
6. If user taps Fix:
   - allow manual edit of total
   - allow manual selection/edit of payment method
   - optional simple note field
7. Save the record

DETECTED TOTAL LOGIC
- Keep it simple for MVP
- Extract numeric values from transcript
- Use the largest plausible number as the likely total
- If confidence is weak or no number is found, set detected_total = null
- Never pretend certainty when unclear

OFFLINE / FAILURE RULES
- Manual entry must remain available
- If transcription fails, user must not be blocked
- If network is unavailable, show a clear fallback path
- Store data locally if needed before sync
- Never trap the user in a dead end

DATA / LEARNING RULES
Even if the V1 UI is simple, structure the data model for future growth.
Create a general transaction schema that works across seller categories.

Include fields like:
- id
- created_at
- source_type (voice/manual)
- raw_audio_url or raw_audio_ref if available
- raw_transcript
- detected_total
- final_total
- payment_method
- notes
- status
- was_edited
- transcription_provider
- parsing_confidence or null
- optional structured fields for future use:
  - items: []
  - item_name
  - quantity
  - unit
  - unit_price
  - line_total
  - optional_attributes
  - color
  - size
  - package
  - customer_name

IMPORTANT:
- It is okay if many of these future fields are null in V1
- Design the schema so V1.1 and V2 can grow into it
- Store raw input + corrected outcome so the dataset becomes valuable over time
