# Report Page — Information Architecture Redesign

> **CRITICAL USER FEEDBACK**: "u r thinking like pos or accountants"
>
> The wireframe organized information by financial categories (Revenue, Expenses, Cash, Transfer). This is accounting language. The shopkeeper doesn't think in these terms. They think: "Did I make money today? How much cash should be in the drawer? Who owes me? Is everything okay?"
>
> **DO NOT rebuild the wireframe with accounting categories.** Rebuild from the shopkeeper's daily reality.

## The Core Problem

The current page has 14 sections, all presented at the same visual weight. Every card has the same border, the same padding, the same font size. The owner's brain has no guide for where to look first.

The fix isn't adding features or removing sections. It's **teaching the eye what matters**.

---

## Content Map

```
REPORT (Shop Check)

├── 1. Executive Summary — "How am I doing?"
│     ├── Health status (adaptive: good / weak / urgent)
│     ├── Profit for the period
│     └── Comparison to prior period
│
├── 2. Money — "Where is the money?"
│     ├── Revenue (sales)
│     ├── Expenses (with category breakdown)
│     ├── Cash expected vs transfer expected
│     ├── Payment method split
│     └── Credit extended vs collected
│
├── 3. Activity — "What happened?"
│     ├── Top selling products
│     ├── Largest sale
│     ├── Biggest expense
│     └── Credit events
│
├── 4. Attention — "What needs action?"
│     ├── Cash closing (if pending)
│     ├── Overdue customers
│     ├── Cash mismatch (if closing done)
│     └── Anomalies (low sales day, high expenses)
│
├── 5. Operations — "Who did what?"
│     ├── Staff performance
│     └── Per-staff breakdown
│
└── 6. Records — "Let me investigate"
      ├── Recent transactions (collapsed)
      ├── Full history (collapsed / separate view)
      ├── Search
      ├── Filter
      └── Export
```

## Information Hierarchy Rules

### Visual weight by importance

| Level | Font | Background | Border | Spacing |
|-------|------|------------|--------|---------|
| ⭐⭐⭐⭐⭐ Executive Summary | 28-32px hero number, 12px label | Colored accent background | Bold, prominent | 16px top, generous |
| ⭐⭐⭐⭐⭐ Money | 16-18px numbers, 10px labels | White cards | Subtle | 12px |
| ⭐⭐⭐⭐ Attention | 14px, colored indicators | Amber/red tinted backgrounds | Colored borders | 12px |
| ⭐⭐⭐ Activity | 13px, compact rows | White, minimal | Hairline | 8px |
| ⭐⭐ Operations | 12px, dense list | White, minimal | Hairline | 6px |
| ⭐ Records | 12px, tucked away | Transparent | None | 4px |

### Progressive disclosure

```
Level 1: "How am I doing?" → Always visible, takes 30% of screen
Level 2: "Where is the money?" → Always visible, takes 25% of screen
Level 3: "What happened?" → Always visible, takes 20% of screen
Level 4: "What needs attention?" → Visible only when items exist
Level 5: "Who did what?" → Collapsed by default (tap to expand)
Level 6: "Let me investigate" → Collapsed by default (tap to expand)
```

## Section-by-Section Design

### 1. Executive Summary (30% of screen)

**Purpose:** One glance answer to "Was today good?"

**Layout:**
```
┌─────────────────────────────────────┐
│  🟢 Business is healthy             │
│                                     │
│  +2,400 ETB profit today            │
│  Better than your average           │
│                                     │
│  ┌─────┐ ┌─────┐ ┌─────┐          │
│  │Sold │ │Spent│ │Profit│          │
│  │8,450│ │4,200│ │2,400 │          │
│  └─────┘ └─────┘ └─────┘          │
│                                     │
│  ▲ 12% vs yesterday                 │
└─────────────────────────────────────┘
```

**Key decisions:**
- The health status is computed, not static. It checks: profit margin, cash match, credit health, sales vs average.
- The 3 sub-cards (Sold, Spent, Profit) are smaller and subordinate to the headline.
- The comparison line adds context without requiring mental math.

**Data:**
- `metrics.totalSold`, `metrics.spentToday`, computed `netProfit`
- Prior period: `buildReportRows` + `computeReportMetrics` for yesterday (today view) or prior week/month
- Health logic: if profit > 0 AND cash matches AND no overdue crisis → healthy. Otherwise → warning/urgent.

### 2. Money (25% of screen)

**Purpose:** Complete financial picture in one place.

**Layout:**
```
┌─────────────────────────────────────┐
│  MONEY                              │
│                                     │
│  Revenue          8,450 ETB         │
│  Expenses        -4,200 ETB         │
│  ─────────────────────────          │
│  Cash expected    6,200 ETB         │
│  Transfer         2,100 ETB         │
│                                     │
│  ┌─── Expense Breakdown ──────┐    │
│  │ Rent       ████░░░  1,800  │    │
│  │ Supplies   ███░░░░  1,200  │    │
│  │ Transport  █░░░░░░    600  │    │
│  └────────────────────────────┘    │
│                                     │
│  Credit extended:  3,200 ETB        │
│  Credit collected: 5,300 ETB        │
│  Net credit flow:  +2,100 ETB       │
└─────────────────────────────────────┘
```

**Key decisions:**
- All money information lives together. No scattering across the page.
- Expense breakdown is inline (not a separate section) because it explains the "Expenses" number above it.
- Credit is framed as a flow (extended vs collected) because that's how the owner thinks about it.

**Data:**
- All from `computeReportMetrics()` — already computed
- Expense breakdown: group `expenseRows` by `item_note || item_name`, sort by total
- Credit flow: `metrics.newDubie` vs `metrics.creditCollected`

### 3. Activity (20% of screen)

**Purpose:** Today's story in plain language.

**Layout:**
```
┌─────────────────────────────────────┐
│  WHAT HAPPENED                      │
│                                     │
│  Top selling: Sugar, Oil, Bread     │
│  Largest sale: 1,200 ETB (wholesale)│
│  Biggest expense: Rent 1,800 ETB   │
│  Credit given to 3 customers       │
│  8 customers paid via Telebirr      │
└─────────────────────────────────────┘
```

**Key decisions:**
- Natural language, not numbers. "Top selling: Sugar" beats "Item: Sugar, Qty: 24, Revenue: 3,200".
- Only 4-5 observations. Not a full breakdown — that lives in Money section.
- Empty observations are hidden (no "No top selling items" text).

**Data:**
- Group `saleRows` by `item_name`, rank by revenue → top 3 items
- Max `amountOf(row)` for sales → largest sale
- Max `amountOf(row)` for expenses → biggest expense
- Count credit rows → "Credit given to N customers"
- Count rows where `payment_provider` includes 'telebirr' → payment observation

### 4. Attention (conditional — only when needed)

**Purpose:** Things that require owner action. Glows when present.

**Layout (when items exist):**
```
┌─────────────────────────────────────┐
│  ⚠ NEEDS ATTENTION                  │
│                                     │
│  🔴 Cash not counted yet            │
│     Expected: 6,200 ETB             │
│     [Count Cash]                    │
│                                     │
│  🟡 Abebe owes 600 ETB (18 days)   │
│  🟡 Tilahun owes 400 ETB (12 days) │
│                                     │
│  🟡 2 credit payments due this week │
└─────────────────────────────────────┘
```

**Layout (when nothing needs attention):**
```
┌─────────────────────────────────────┐
│  ✅ Everything looks good.           │
│     Nothing needs your attention.   │
└─────────────────────────────────────┘
```

**Key decisions:**
- Uses colored indicators (red/amber/green) for urgency.
- "Cash not counted" is always first if pending — it's the most time-sensitive.
- Overdue customers listed by amount (highest first).
- When nothing is wrong, the section shrinks to a single reassuring line.

**Data:**
- Cash closing status: check `closings` array for today
- Overdue customers: `enrichedCustomerSummaries` filtered for `has_overdue`
- Upcoming due dates: credits with `due_date` within next 7 days
- Anomaly detection: sales < 50% of 7-day average → flag

### 5. Operations (collapsed by default)

**Purpose:** Staff accountability. Only owners with staff care.

**Layout (collapsed):**
```
┌─────────────────────────────────────┐
│  STAFF PERFORMANCE          ▶ 3 staff│
└─────────────────────────────────────┘
```

**Layout (expanded):**
```
┌─────────────────────────────────────┐
│  STAFF PERFORMANCE           ▼      │
│                                     │
│  Abebe    3,200 ETB  12 sales      │
│  Feven    2,100 ETB   8 sales      │
│  Owner    3,150 ETB  15 sales      │
│                                     │
│  Filter by staff member ▼           │
└─────────────────────────────────────┘
```

**Key decisions:**
- Collapsed by default because most solo shopkeepers don't need this.
- When expanded, shows the same staff breakdown that exists today.
- The "Filter by staff" dropdown moves here from its current prominent position.

**Data:**
- `buildStaffReportRows(reportRows)` — already computed at line 760

### 6. Records (collapsed by default)

**Purpose:** Investigation tools. Deep dive when needed.

**Layout (collapsed):**
```
┌─────────────────────────────────────┐
│  RECORDS                   ▶ 47 items│
└─────────────────────────────────────┘
```

**Layout (expanded):**
```
┌─────────────────────────────────────┐
│  RECORDS                    ▼       │
│                                     │
│  [Search transactions...]    🔍    │
│                                     │
│  Sugar sale     320 ETB  Cash  9:32 │
│  Oil sale       180 ETB  Telebirr   │
│  Credit: Abebe  600 ETB  Credit     │
│  ...                                │
│                                     │
│  [Load more]                        │
│                                     │
│  ─────────────────────              │
│  Export  │  Full History            │
└─────────────────────────────────────┘
```

**Key decisions:**
- Search, filter, export, and full history all live here — they're investigation tools, not summary information.
- The 47 items count tells the owner how much data is behind the collapse.
- Search is inside Records, not at the top of the page.

**Data:**
- `filteredTransactions` — already computed
- Search, filter, export logic — already exists

### Phase 2: Visual Hierarchy (1-2 weeks)

**Key visual changes:**

1. **Story Card** gets the most visual weight:
   - Background: subtle gradient (green when healthy, amber when warning, red when urgent)
   - Hero number: 28-32px
   - Takes full width, generous padding

2. **Money Section** uses compact financial layout:
   - Revenue/Expenses as a simple two-column top row
   - Cash/Transfer as a second row
   - Credit as a third row with flow arrows
   - Expense breakdown as inline horizontal bars (pure CSS)

3. **Attention Section** uses colored indicators:
   - Red background tint for urgent items
   - Amber for warnings
   - Green for "all good" state
   - The [Count Cash] button is prominent when pending

4. **Operations + Records** use minimal styling:
   - Hairline borders
   - Smaller font sizes
   - Collapsed by default with chevron indicators
   - Expand animation (already exists in current `Section` component)

5. **Remove visual noise:**
   - Remove the Dashboard Insight Strip (redundant with Story Card)
   - Remove the fixed bottom action bar (tools belong in Records section)
   - Remove the KPI detail sheet (inline explanations instead)

### Phase 3: Adaptive Behavior (1 week)

**The Story Card changes based on reality:**

Compute a health score from:
- Profit margin (positive = good)
- Cash match (if closing done: within 5% = good)
- Credit health (overdue ratio < 20% = good)
- Sales vs 7-day average (within 80% = good)

Each factor contributes to a composite status:
- 4/4 good → 🟢 "Business is healthy"
- 3/4 good → 🟡 "Business is mostly healthy" + specific note
- 2/4 or less → 🔴 "Needs attention" + specific issues

**The Attention section adapts:**
- Cash not counted → always shown first if pending
- Overdue customers → shown by severity (oldest/most first)
- Cash mismatch → shown after closing with variance
- Low sales → shown if < 50% of average

## Verification

1. **Visual hierarchy test:** Open the page on a 360px mobile screen. Can you identify the most important number within 2 seconds? If not, the hierarchy isn't working.

2. **Progressive disclosure test:** Tap through each collapsed section. Does expanding Operations feel natural? Does Records feel like a deep-dive tool, not a primary view?

3. **Adaptive story test:** Create scenarios:
   - Good day (profit positive, cash matches, no overdue) → should show 🟢
   - Bad day (loss, cash mismatch) → should show 🔴 with specific issues
   - No transactions today → should show a neutral/informational state

4. **Empty state test:** A shop with no staff, no credit, no expenses — what does the page look like? It should still be useful, not full of empty cards.

5. **Amharic test:** All labels, observations, and health messages render correctly in Amharic.

6. **Performance test:** On a low-end Android phone with 5000+ transactions, does the page load and scroll smoothly? Memoize all heavy computations.
