# Fix: App Treats User as New on Refresh / Close-Reopen

## Root Cause

The app has **two completely separate auth systems that are never connected**:

| System | User Table | User ID | Token | Used By |
|--------|-----------|---------|-------|---------|
| OTP Auth (`/api/auth/verify`) | Postgres `users` (serial int) | numeric | JWT | Sync engine, customer ledger, backup/restore |
| Identity System (`/api/shops`) | In-memory store (UUID) | string | device_token (random UUID) | Staff events, activity feed |

**The problem:** The owner onboarding flow (`OnboardingScreen.jsx`) and staff join flow (`StaffJoinScreen.jsx`) both use the identity system to create shops/join. They call `setIdentity()` which saves the `device_token` to IndexedDB. But they **never call `setAuthToken()`** to save a JWT to `db.settings.gebya_auth_token`.

The sync engine (`syncEngine.js:264-266`) reads from `getAuthToken()` → `db.settings.gebya_auth_token`. Since this key is never set, the sync engine sets `status = 'unauthenticated'` and never syncs.

**On page refresh:**
1. `loadData()` reads identity from IndexedDB → `shopProfile` gets populated → app shows the shop UI (not onboarding)
2. Sync engine tries to sync → `getAuthToken()` returns null → `status = 'unauthenticated'`
3. All server-side data is unreachable → the app behaves as if it's a fresh install

Additionally, `useAuthStore` (which has the `init()` method that would validate the token on boot) is **dead code** — never imported or called anywhere.

## Why a simple "exchange device_token for JWT" won't work

The identity system's user IDs are UUIDs (string). The OTP auth system's user IDs are serial integers. A device-token-to-JWT endpoint would need to map between these two incompatible ID systems. The identity system also uses an in-memory store, not Postgres.

## Plan

### Step 1: Frontend — Save JWT via OTP auth after onboarding (owner flow)

**File:** `artifacts/gebya/src/components/OnboardingScreen.jsx`

After `setIdentity(identity)` succeeds (line 133), trigger the OTP auth flow to get a JWT:

1. If the user entered a phone number during onboarding, call `requestOtp(fullPhone)` to send an OTP
2. Show an OTP input screen (reuse the existing OTP verification pattern from `AuthGate.jsx`)
3. On OTP verify, call `verifyOtp(fullPhone, otp)` to get a JWT
4. Call `setAuthToken(token)` to save the JWT to IndexedDB
5. If no phone was entered, skip OTP (sync will remain unauthenticated until phone is added in settings)

**File:** `artifacts/gebya/src/components/AuthGate.jsx` — This component already has the full OTP flow (requestOtp → verifyOtp → setAuthToken). Extract the OTP input UI into a reusable component or import it directly into OnboardingScreen.

### Step 2: Frontend — Save JWT via OTP auth after staff join

**File:** `artifacts/gebya/src/components/StaffJoinScreen.jsx`

After `persistIdentity(result)` succeeds (line 112), trigger OTP auth:

1. If the staff member entered a phone number, call `requestOtp(phone)` and show OTP input
2. On verify, call `setAuthToken(token)` to save the JWT
3. If no phone was entered, skip (sync will be unauthenticated)

### Step 3: Frontend — Wire up `useAuthStore` for boot-time auth validation

**File:** `artifacts/gebya/src/components/AppShell.jsx`

In `loadData()`, after identity is loaded from IndexedDB (line 621), call `useAuthStore.getState().init()` to validate the stored JWT against the server. If the token is expired/invalid, clear it and fall back gracefully.

**File:** `artifacts/gebya/src/App.jsx`

Add an auth initialization useEffect that calls `useAuthStore.getState().init()` on mount, before `AppShell` renders. Or gate rendering of `AppShell` on `useAuthStore`'s `checked` state.

### Step 4: Frontend — Fix sync engine to handle token re-acquisition

**File:** `artifacts/gebya/src/utils/syncEngine.js`

On 401/403 in `sync()` (line 285-287), before clearing the token, attempt to re-acquire a JWT:
1. Read the device_token from `getIdentity()`
2. If device_token exists, try to re-exchange for a JWT (via OTP or a future backend endpoint)
3. If re-acquisition succeeds, retry the sync
4. Only clear the token if re-acquisition also fails

### Step 5 (optional, future): Backend — Add device-token-to-JWT bridge

**File:** `artifacts/api-server/src/routes/auth.ts`  
**File:** `lib/db/src/schema/shops.ts`

Add `POST /api/auth/exchange-device-token` that:
1. Accepts `{ device_token }` in the body
2. Hashes the token, looks up device in the Postgres `devices` table (from shops.ts schema)
3. Finds staff → user chain in Postgres
4. Issues a JWT via `signJwt(userId)`
5. Returns `{ token, user, role, permissions }`

This requires the identity routes to also write to Postgres (not just the in-memory store). The identity routes currently only use the in-memory `store` — they need to also insert into the Postgres tables. This is a larger change that should be done as a separate task.

## Verification

1. **Fresh owner flow:** Create a shop with phone → enter OTP → close browser → reopen → app should load the shop AND sync should work
2. **Refresh test:** After creating a shop with OTP, refresh the page → same result
3. **Staff join flow:** Join a shop with phone → enter OTP → close/reopen → app should load correctly with sync
4. **No phone flow:** Create a shop without phone → app works locally but sync is unauthenticated → adding phone in settings triggers OTP → sync activates
5. **Expired token:** After 30 days (or manually clear IndexedDB auth token), `authStore.init()` detects the expired token → sync fails → user re-authenticates via phone/OTP
6. **Offline scenario:** If OTP request fails (offline), the app should still show local data and prompt for OTP when online
