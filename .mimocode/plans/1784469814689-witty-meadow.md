# Gebya Platform — End-to-End Review

## Platform Overview
Gebya (ገበያ) is a voice-first, mobile-first PWA for Ethiopian micro-retailers to replace paper notebooks for tracking sales, expenses, credit (Merro), and supplier payments. Built as a pnpm monorepo with three packages:

| Package | Stack | Role |
|---------|-------|------|
| `artifacts/gebya/` | React 18 + Vite + Tailwind v4 + Dexie.js (IndexedDB) + Zustand | Frontend PWA |
| `artifacts/api-server/` | Express 5 + Drizzle ORM + PostgreSQL + JWT | Backend API |
| `lib/` | db, api-zod, api-spec, api-client-react | Shared libraries |

**Deployment**: Vercel (frontend + serverless), GitHub for source control.

## CRITICAL ISSUES (Must Fix Before Launch/Scale)

### 1. API Keys and Secrets Exposed in .env File
**File**: `.env` (lines 9-15)
**Severity**: CRITICAL — Security

The `.env` file contains plaintext API keys and VAPID private keys:
```
SMS_ETHIOPIA_API_KEY="T2XDT3ML7CO1KF4GLKY8RTYPVOZTEQJ6:1369"
VAPID_PRIVATE_KEY=9v8cIFo7EGlhAsO5CnVOc6IYFbFnu8pdsHferdnuFTs
```

While `.gitignore` includes `.env`, these secrets appear to exist in the repo. If this repo is public or ever was, these keys are compromised. **Action needed**:
- Rotate all exposed keys immediately
- Move to Vercel environment variables (not checked into repo)
- Add `.env` to a `.env.example` with placeholder values only
- Audit git history for leaked secrets

### 2. JWT Token Has 30-Day Expiry with No Refresh Mechanism
**File**: `artifacts/api-server/src/routes/auth.ts:18`
**Severity**: HIGH — Security

```typescript
const JWT_EXPIRES_IN = "30d";
```
A 30-day JWT with no refresh token mechanism means:
- Stolen tokens are valid for a full month
- No way to revoke access without changing JWT_SECRET (which would invalidate ALL users)
- No session management

**Action needed**: Implement short-lived access tokens (15 min) + refresh tokens with rotation.

### 3. OTP Uses Math.random() — Not Cryptographically Secure
**File**: `artifacts/api-server/src/routes/auth.ts:27-29`
**Severity**: HIGH — Security

```typescript
function generateOtp() {
  return String(Math.floor(100000 + Math.random() * 900000));
}
```
`Math.random()` is predictable. An attacker who can predict the PRNG can forge OTPs. Should use `crypto.randomInt()`.

### 4. No Rate Limiting on Auth Endpoints
**File**: `artifacts/api-server/src/routes/auth.ts`
**Severity**: HIGH — Security

The `/api/auth/otp` and `/api/auth/verify` endpoints have no rate limiting. An attacker can:
- Flood OTP requests (SMS cost attack / spam)
- Brute-force 6-digit OTPs (1M combinations, no lockout beyond per-record limit)

**Action needed**: Add IP-based rate limiting on auth routes (e.g., 5 OTP requests per phone per 10 min, 10 failed verifies per IP per hour).

### 5. Telegram Webhook Secret is Optional
**File**: `artifacts/api-server/src/routes/telegram.ts:558-562`
**Severity**: HIGH — Security

```typescript
const expectedSecret = process.env.TELEGRAM_WEBHOOK_SECRET?.trim();
if (expectedSecret && receivedSecret !== expectedSecret) {
```
If `TELEGRAM_WEBHOOK_SECRET` is not set, the webhook is **completely unauthenticated** — anyone can POST fake updates to your Telegram bot endpoint.

**Action needed**: Make `TELEGRAM_WEBHOOK_SECRET` required; reject requests if not configured.

## HIGH-PRIORITY ISSUES

### 6. Massive AppShell.jsx God Component (~1200+ lines)
**File**: `artifacts/gebya/src/components/AppShell.jsx`
**Severity**: HIGH — Maintainability

This single component:
- Manages 50+ state variables
- Contains 30+ useEffect hooks
- Handles all business logic (transactions, customers, suppliers, staff, sync, telegram, analytics)
- Contains inline components (AuthRequiredPrompt, TrustCard, OfflineStatusStrip)

This is unmaintainable at scale. Any change risks cascading regressions.

**Action needed**: Decompose into feature-specific providers/hooks (e.g., `useTransactionManager`, `useCustomerManager`, `useSyncManager`).

### 7. Dexie Database Has 24 Schema Versions with No Cleanup
**File**: `artifacts/gebya/src/db.js`
**Severity**: MEDIUM-HIGH — Technical Debt

24 sequential schema versions (v1-v24) with inline migration code. Some versions are no-ops (v1-v2 are identical). The migration at v12 runs a complex `migrateLegacyCredits` that reads from tables already dropped in later versions. Risk of broken migrations for users on very old versions.

**Action needed**: 
- Test all migration paths (v1→v24) end-to-end
- Consider collapsing old no-op versions
- Add migration tests (some exist but coverage is unclear)

### 8. Dual Schema Definition (Frontend Dexie vs Backend Drizzle)
**Severity**: MEDIUM-HIGH — Data Integrity Risk

The frontend defines its schema in `db.js` (Dexie/IndexedDB) and the backend defines its own in `lib/db/schema/` (Drizzle/PostgreSQL). These are manually kept in sync. There is NO shared type definition or code generation between them. Any drift between frontend and backend schemas causes silent data corruption during sync.

**Action needed**: Generate shared schema types or add integration tests that validate schema parity.

### 9. No Input Validation on Frontend Transaction Forms
**Severity**: MEDIUM — Data Integrity

Looking at `AppShell.jsx:handleAddTransaction`, transaction data is passed directly to Dexie without validation. The backend uses Zod schemas, but the frontend does not validate amounts, types, or required fields before persisting locally. Users can create transactions with negative amounts, invalid types, or empty required fields.

**Action needed**: Add Zod validation on the frontend before local persistence (the api-zod library already exists).

### 10. Stale Chunk Self-Heal Can Cause Infinite Reload Loop
**File**: `artifacts/gebya/src/components/AppShell.jsx:207-230` and `ErrorBoundary.jsx`
**Severity**: MEDIUM — Reliability

Both AppShell and ErrorBoundary have chunk-reload recovery logic. The AppShell uses a sessionStorage flag, and ErrorBoundary uses a different key (`gebya_chunk_reload_attempted`). If both fire simultaneously, they could race. Also, the `lazyWithRetry` returns `new Promise(() => {})` which permanently hangs the render if the reload fails.

**Action needed**: Unify the recovery mechanism and add a maximum retry count with a user-visible fallback.

## MEDIUM-PRIORITY ISSUES

### 11. No Offline Queue for Failed Sync Pushes
**File**: `artifacts/gebya/src/utils/syncEngine.js`
**Severity**: MEDIUM — Data Reliability

The sync engine has retry logic with exponential backoff for API calls, but failed pushes are not persisted to a local queue. If the user goes offline mid-sync, those changes may be lost on next sync if the in-memory state is cleared.

**Action needed**: Persist pending sync operations to IndexedDB and retry on reconnect.

### 12. `@ts-nocheck` on Main Server Entry Point
**File**: `artifacts/api-server/src/main.ts:1`
**Severity**: MEDIUM — Code Quality

```typescript
// @ts-nocheck
```
Disabling TypeScript checking on the main entry point defeats the purpose of using TypeScript. This likely hides type errors.

**Action needed**: Remove `@ts-nocheck` and fix underlying type issues.

### 13. Hardcoded CORS Origin in vercel.json
**File**: `artifacts/api-server/vercel.json:17`
**Severity**: MEDIUM — Configuration

```json
"Access-Control-Allow-Origin": "https://gebya-notebook-addis-gebya.vercel.app"
```
This is also handled in the Express CORS middleware, creating a double-CORS configuration. The vercel.json header is static and can't adapt to preview deployments.

**Action needed**: Remove the vercel.json CORS headers and rely solely on the Express middleware.

### 14. Missing CSRF Protection
**Severity**: MEDIUM — Security

The API uses JWT Bearer tokens (stateless), which provides some CSRF protection, but:
- The sync push endpoint accepts data from any authenticated user
- No CSRF tokens on state-changing POST endpoints
- Cookie-based auth would be vulnerable

**Action needed**: For current Bearer token approach, ensure no cookies are used for auth. Consider adding CSRF tokens if switching to cookie-based sessions.

### 15. Backup Snapshot Stores Raw Payload as String
**File**: `artifacts/api-server/src/routes/backup.ts:76`
**Severity**: MEDIUM — Security/Performance

Backups are stored as raw JSON strings up to 10MB. This means:
- No encryption of potentially sensitive financial data
- Large payloads can cause slow API responses
- No compression

**Action needed**: Encrypt backup payloads, add compression, and consider streaming for large backups.

### 16. Analytics Data Stored Unencrypted in IndexedDB
**File**: `artifacts/gebya/src/db.js:480` (analytics table)
**Severity**: MEDIUM — Privacy

Session counts, streaks, feature usage, and days active are stored in plain IndexedDB. While not financial data, this behavioral data could reveal business patterns if the device is compromised.

**Action needed**: Consider encrypting analytics data or storing it in a more secure location.

### 17. No Content Security Policy (CSP)
**Severity**: MEDIUM — Security

Helmet is configured but likely without a strict CSP. The PWA loads scripts from multiple origins (Vite chunks, workbox, Sentry). Without CSP, XSS vulnerabilities can load arbitrary scripts.

**Action needed**: Add a strict CSP header via Helmet configuration.

## LOW-PRIORITY / UX ISSUES

### 18. Error Messages Mix English and Amharic Inconsistently
**Severity**: LOW — UX

Looking at error toasts in `AppShell.jsx`:
```javascript
fireToast({
  message: lang === 'am'
    ? `ouis በዚህ ወር ${entitlements.max_transactions_per_month} ግብይቶች በቂ ናቸው። Plus ወደ አድሶ ያዝ።`
    : `You've reached the ${entitlements.max_transactions_per_month} transaction limit this month. Upgrade to Plus to continue.`,
});
```
The Amharic message starts with "ouis" (a typo/artifact). Some messages are hardcoded in English even when `lang === 'am'`.

### 19. No Loading States for Critical Operations
**Severity**: LOW — UX

Customer deletion, transaction editing, and backup operations don't show loading indicators. Users may tap multiple times, causing duplicate operations.

### 20. Voice Input Feature Appears Removed
**Severity**: LOW — Feature Gap

The spec prominently features voice-first transaction entry, but the database migration v22 explicitly removes voice-related fields:
```javascript
delete record.raw_transcript;
delete record.detected_total;
delete record.transcription_provider;
```
This suggests voice was built and then removed. The spec still describes it as a core MVP feature. Either the spec needs updating or voice needs to be re-implemented.

### 21. Triaging Branches Are Duplicated
**Severity**: LOW — Repo Hygiene

The repo contains `_triage-master/` and `_triage-feature/` which appear to be full copies of the codebase. These add significant repo bloat and can cause confusion.

**Action needed**: Remove or gitignore triage directories.

## WHAT'S WORKING WELL

1. **Offline-first architecture** — Dexie.js with 24-version schema evolution shows thoughtful data layer design
2. **Privacy model** — Money values hidden by default, auto-hide, privacy toggle is a great UX decision for the target market
3. **Bilingual support** — Amharic/English toggle with proper RTL consideration
4. **Telegram integration** — Well-designed webhook system with bilingual bot responses
5. **PWA configuration** — Proper caching headers, service worker registration, install prompts
6. **Sync engine** — Version-based conflict detection with exponential backoff retry
7. **Progressive disclosure** — Advanced fields hidden, keeping forms simple for the 80% case
8. **Staff/RBAC system** — Role-based permissions with owner/cashier/viewer roles
9. **Error boundaries** — Auto-recovery from chunk load errors, user-friendly error UI
10. **Cultural design** — Ethiopian calendar, Birr formatting, credit-focused UX
